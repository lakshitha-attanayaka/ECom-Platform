# ADR-001: Subdomain-Based Tenant Context Resolution

## Status
Approved

## Context
Under our white-label multi-tenant e-commerce architecture, we must dynamically resolve the tenant context for each incoming HTTP request before any domain logic, authentication, or database operations execute. The platform must provision dedicated subdomains for merchants (e.g., `tenant-a.localhost` or `tenant-b.platform.com`) and customize storefronts accordingly.

## Decision
We implement a custom request-scoped middleware (`TenantContextMiddleware`) in the ASP.NET Core pipeline that:
1. Extracts the Host header from the request.
2. Parses the first segment (subdomain slug) of the Host header (e.g., `tenant-a` from `tenant-a.localhost`).
3. Queries the central PostgreSQL registry (`saasdb`) using `TenantDbContext` to find the matching active tenant.
4. Memory-caches the resolved tenant entity using `IMemoryCache` with a 5-minute sliding expiration (or 1-minute expiration on cache miss) to avoid database hits on every request.
5. Injects the tenant details into a request-scoped `ITenantContext` service.
6. Enriches OpenTelemetry traces and activity tags with the `tenant.id` and `tenant.slug`.
7. Intercepts and short-circuits requests for invalid or inactive tenants with a structured HTTP 404 response.

## Alternatives Considered

### 1. JWT Claim-Based Tenant ID (Rejected)
* **Reasoning:** In a multi-realm Keycloak setup, the application must already know which Keycloak realm to use *before* prompting the user for authentication. Relying on a tenant ID claim inside the JWT creates a circular dependency: we need the tenant context to configure the OIDC authority realm, but we would need the authenticated JWT to resolve the tenant context.

### 2. Custom HTTP Header (Rejected)
* **Reasoning:** Sourcing the tenant identifier from a client-supplied header (e.g., `X-Tenant-Id`) is a major security risk. Users could modify or forge the header to access other tenants' data. Subdomain-based routing is bound to the DNS/TLS layer and is much more secure, transparent, and user-friendly.

### 3. Path-Based Routing (e.g., `platform.com/tenant-a`) (Rejected)
* **Reasoning:** Makes white-labeling and custom domain mapping difficult. Subdomains align directly with merchant expectations for independent storefronts and easily support future custom DNS CNAMES.

## Consequences
* **Service Dependency:** All domain APIs and BFF services must depend on the scoped `ITenantContext` to retrieve the current tenant context.
* **Propagation Delay:** Because of memory caching, changes to a tenant's subscription status or database entity will take up to 5 minutes to propagate across all running BFF instances. For cache misses, a 1-minute TTL prevents database denial-of-service from requests targeting non-existent subdomains.
* **Global Query Filters:** Downstream database repositories must configure EF Core Global Query Filters bound to `ITenantContext.TenantId` to guarantee tenant isolation at the SQL generation level.
* **Rate Limiting:** Per-tenant rate limiters must be partitioned using `ITenantContext.TenantId` to ensure a single tenant's traffic spike does not impact other tenants.
